# 微信 4.1.11 独立读取链路技术方案

## 1. 背景

当前 Windows 桌宠的自动链路已经具备以下能力：

1. 桌宠启动后自动后台运行
2. 自动检测微信进程
3. 自动提取数据库密钥
4. 自动导出聊天数据
5. 自动上传到服务端

其中，现有解密读取链路分为两段：

- 密钥提取：`WeFlowKeyBridge.exe` + `wx_key.dll`
- 数据库读取：`WeFlowWcdbBridge.exe` + `wcdb_api.dll`

实际运行中，`wcdb_api.dll` 在初始化阶段返回 `-1006`。结合现有分析，可合理判断其包含在线授权/环境校验，因而不适合作为桌宠正式量产链路的核心依赖。

结论：

- `WeFlowKeyBridge.exe` 可以继续保留
- `WeFlowWcdbBridge.exe` 不应继续作为正式方案
- 需要建设自有的独立数据库读取链路

## 2. 目标

建设一条不依赖 `wcdb_api.dll` 的本地读取方案，保持桌宠现有的全自动体验不变：

1. 双击桌宠即可启动
2. 后台自动检测微信
3. 自动提取密钥
4. 自动读取聊天数据库
5. 自动生成 `chatlog_export.json`
6. 自动沿用现有上传链路推送到服务端

## 3. 非目标

本方案当前不承诺以下事项：

1. 不承诺“纯 C# 一定能解开微信 4.1.11 数据库”
2. 不承诺首期覆盖图片、视频、语音二进制内容导出
3. 不承诺首期替换现有所有桌宠/后端逻辑
4. 不继续依赖或深入绑定 `wcdb_api.dll`

原因很简单：当前真正未确认的是微信 4.1.11 数据库的实际页级加密格式，而不是语言选型。

## 4. 当前已确认事实

### 4.1 可以保留的部分

`WeFlowKeyBridge.exe` 已经具备独立价值：

- 可直接加载 `wx_key.dll`
- 可自动定位 `Weixin.exe` / `WeChat.exe`
- 可获取 `dbKey`
- 可返回 `wxid`
- 可返回状态消息

因此，“取密钥”这一层不需要推倒重来。

### 4.2 需要淘汰的部分

`WeFlowWcdbBridge.exe` 当前依赖：

- `wcdb_api.dll`
- `WCDB.dll`
- `SDL2.dll`

而实际失败点发生在：

1. `InitProtection()`
2. `wcdb_init()`

并非简单路径错误或缺 DLL 问题，因此继续围绕 `wcdb_api.dll` 修补不适合作为正式主线。

### 4.3 现有自动链路不需要重做

当前总入口仍然可以保留：

- `windows/wx_decrypt.py`

现有桌宠上传侧也可以保留：

- `windows/WeChatMonitor.cs`

只要新的读取器继续输出：

- `chatlog_export.json`
- `decrypt_meta.json`

那么服务端和桌宠上传部分都不需要大改。

## 5. 总体方案

新链路调整为：

```text
桌宠启动
-> WeChatMonitor.Start()
-> wx_decrypt.py
-> WeFlowKeyBridge.exe
-> 获取 dbKey / wxid
-> IndependentWcdbReader.exe
   -> DbProbe
   -> WcdbDecryptor
   -> SqliteExporter
-> 生成 chatlog_export.json / decrypt_meta.json
-> 继续走现有上传链路
```

核心思想：

- 保留“取密钥”
- 重写“读数据库”
- 保持“输出契约”和“上传契约”不变

## 6. 核心组件设计

建议新增目录：

- `windows/IndependentWcdbReader/`

建议新增文件：

- `windows/IndependentWcdbReader/Program.cs`
- `windows/IndependentWcdbReader/DbProbe.cs`
- `windows/IndependentWcdbReader/WcdbDecryptor.cs`
- `windows/IndependentWcdbReader/SqliteExporter.cs`
- `windows/IndependentWcdbReader/Models.cs`
- `windows/IndependentWcdbReader/IndependentWcdbReader.csproj`

### 6.1 Program.cs

职责：

1. 解析命令行参数
2. 调度 `probe` / `decrypt` / `export` 三种模式
3. 输出标准 JSON 结果

建议参数：

```text
--account-dir
--wxid
--key
--session-db
--message-db
--output-json
--output-sqlite
--mode
```

### 6.2 DbProbe.cs

职责：

1. 读取数据库文件头
2. 检测是否存在标准 `SQLite format 3`
3. 推测页大小
4. 观察第一页和后续页结构
5. 分析是否存在 page trailer / salt / mac 区域

输出内容建议包括：

- 文件大小
- 前 64 / 256 字节十六进制
- 推测页大小
- 是否出现标准 SQLite 头
- 第 1 页与第 N 页结构差异
- 初步格式判断

`DbProbe` 的目标不是解密，而是回答一个问题：

`这个数据库是否像标准 SQLite + 页级加密库。`

### 6.3 WcdbDecryptor.cs

职责：

1. 基于 `dbKey` 和探测结果构建解密流程
2. 按页读取数据库
3. 逐页尝试解密
4. 输出临时明文 SQLite 文件

这里的关键注意点：

- 不能直接假设“整文件 AES 解密”
- 不能直接假设“只要是 64 位 hex key 就能直接用”
- 需要先根据 `DbProbe` 的结果决定：
  - 是否有 KDF
  - 是否存在 salt
  - 是否有页尾校验
  - 第一页是否需要特殊处理

首期目标非常明确：

`先让 session.db 成功还原为可被 SQLite 打开的临时库。`

### 6.4 SqliteExporter.cs

职责：

1. 打开临时明文 SQLite
2. 读取 `sqlite_master`
3. 识别会话表、消息表
4. 导出兼容现有服务端的 JSON 结构

首期导出目标：

1. 会话列表
2. 最近消息
3. 消息基础字段
4. 发送/接收方向
5. 时间戳
6. 联系人基础标识

首期不强求：

1. 图片二进制
2. 视频二进制
3. 语音二进制
4. 完整富媒体还原

## 7. 分阶段实施计划

### 阶段 1：DbProbe

目标：

- 对真实 `session.db` / `message_0.db` 样本完成结构探测

输入：

- `db_storage/session/session.db`
- `db_storage/message/message_0.db` 或 `biz_message_0.db`
- `dbKey`

输出：

- 一份结构分析结果
- 对页大小、文件头、页尾结构的初步结论

验收标准：

1. 能输出稳定的探测 JSON
2. 能确认是否存在标准 SQLite 页特征
3. 能给出下一步是否值得继续纯托管实现的判断

### 阶段 2：可行性判定

目标：

- 决定是否继续推进“纯 C# 页解密”

判定条件：

1. 若格式可被稳定描述为“标准 SQLite + 可复现页级加密”
   - 进入阶段 3A
2. 若格式明显存在复杂私有变体，且纯托管实现风险高
   - 进入阶段 3B

### 阶段 3A：纯托管解密器

目标：

- 用 C# 实现页级解密，输出临时 SQLite

验收标准：

1. 能成功读出 `sqlite_master`
2. 能列出表名
3. 能读出至少一条会话记录
4. 能稳定导出基础消息

### 阶段 3B：自有原生读取器兜底

目标：

- 如果纯托管方案不可行，则建设自有本地原生读取器

说明：

- 这里的“原生读取器”是兜底技术路线
- 目的仍然是不再依赖 `wcdb_api.dll`
- 仍然保持本地静默、自动运行、对现有上传链路透明

### 阶段 4：接回现有桌宠链路

目标：

- 替换 `wx_decrypt.py` 中的 `WeFlowWcdbBridge.exe`
- 改为调用 `IndependentWcdbReader.exe`

保持不变的部分：

- 目录识别
- 微信进程检测
- 事件上报
- `chatlog_export.json`
- `decrypt_meta.json`
- `WeChatMonitor.cs` 上传逻辑

## 8. 与现有代码的关系

### 8.1 保留

- `windows/WeFlowKeyBridge/Program.cs`
- `windows/wx_decrypt.py`
- `windows/WeChatMonitor.cs`

### 8.2 淘汰

- `windows/WeFlowWcdbBridge/Program.cs`

### 8.3 替换点

`wx_decrypt.py` 当前负责：

1. 找到 v4 数据目录
2. 拿到 `wxid`
3. 调用桥接程序
4. 生成导出文件

未来改动点只有一处：

- 将“调用 `WeFlowWcdbBridge.exe`”替换为“调用 `IndependentWcdbReader.exe`”

这意味着桌宠主进程不需要做架构级重写。

## 9. 风险与判断

### 9.1 最大风险

最大风险不是 C#，而是格式未知。

当前尚未确认：

1. `dbKey` 是否可直接用于库解密
2. 是否存在密钥派生步骤
3. 是否存在页尾校验
4. 第一页是否特殊加密
5. 微信 4.1.11 是否使用了标准 WCDB/SQLite codec 变体

### 9.2 为什么不直接承诺纯 C#

因为目前缺的不是编码时间，而是格式证据。

没有 `DbProbe` 结果之前，任何“纯 C# 一定能解”的结论都过早。

### 9.3 为什么仍值得继续

因为当前链路里最关键的前置条件已经具备：

1. 能定位真实数据目录
2. 能提取密钥
3. 能识别当前登录微信
4. 能复用现有导出和上传协议

也就是说，真正未知的只剩数据库页格式这一层。

## 10. 验收标准

最终以以下结果作为方案成功标准：

1. 双击桌宠后仍然全自动后台运行
2. 不再依赖 `wcdb_api.dll`
3. 能稳定提取当前登录微信的 `dbKey`
4. 能稳定生成 `chatlog_export.json`
5. 现有监控页面能看到完整事件链路
6. 现有后端接口无需改协议即可继续接收

阶段性成功标准：

1. `DbProbe` 能稳定输出结构结论
2. `WcdbDecryptor` 能还原 `session.db`
3. `SqliteExporter` 能导出至少一批基础消息

## 11. 结论

本项目接下来的正确主线不是继续围绕 `wcdb_api.dll` 修补，而是：

1. 保留 `WeFlowKeyBridge.exe`
2. 淘汰 `WeFlowWcdbBridge.exe`
3. 新建 `IndependentWcdbReader.exe`
4. 先做探测，再决定是否走纯托管解密
5. 最终接回现有 `wx_decrypt.py` 与桌宠上传链路

一句话总结：

`先证明格式，再实现读取；先替换数据库读取层，不动桌宠和后端主链路。`
