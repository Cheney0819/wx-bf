# 微信数据库解密的部分成功设计

## 目标

让已经通过验证的聊天记录数据库继续完成导出与上传；联系人、收藏等辅助数据库的失败只产生可诊断告警，不再让整轮聊天记录同步失败。

## 范围与边界

- `message` 与 `session` 是必需数据库。任何一个未完成密钥验证、稳定快照、解密或 SQLite 完整性校验时，本轮不得导出或上传聊天记录。
- 联系人、收藏及其他数据库是辅助数据库。其失败不得阻断消息导出和上传。
- 不修改密钥；诊断事件不上传密钥、数据库文件或聊天内容。
- 不改变后端部署，也不推送 Android 或后端项目到远程仓库。

## 解密结果模型

解密器将为每个数据库记录：相对路径、必需性、快照状态、密钥匹配状态、解密状态和 SQLite 完整性状态。

失败原因统一归类为：

- `key_unmatched`：未找到能通过数据库首页 HMAC 验证的密钥；
- `snapshot_unstable`：微信持续写入，有限次重试后仍无法取得一致快照；
- `decrypt_failed`：HMAC 校验或页面解密失败；
- `integrity_failed`：解密产物无法通过 SQLite `quick_check`；
- `source_read_failed`：源数据库或其 WAL/SHM 无法读取。

汇总结果包含必需库成功数、辅助库成功数、辅助库失败条目，以及是否允许导出消息。

## 处理流程

1. 定位 V4 数据目录并从微信进程内存匹配数据库密钥。
2. 为候选数据库创建稳定快照，并解密后运行 SQLite 完整性校验。
3. `message` 或 `session` 失败时，发出包含具体库和分类原因的失败事件并停止本轮上传。
4. 两个必需库成功时，允许导出 `chatlog_export.json` 并继续现有增量上传流程。
5. 辅助库失败时，继续上传聊天记录，同时上报结构化告警；页面显示“聊天记录已上传，辅助库失败”。
6. 临时明文产物仍在本轮结束时清理。

## 可观测性与隐私

现有解密完成事件新增必需库/辅助库的统计和每个失败库的分类原因。事件不得包含密钥、数据库字节、聊天正文或绝对用户目录；仅包含脱敏后的相对数据库路径和状态。

## 测试

必须新增并先失败后通过的自动化测试：

1. `message` 与 `session` 成功、辅助库失败时，汇总结果允许消息导出。
2. `message` 失败时，汇总结果拒绝消息导出并返回 `key_unmatched`、`snapshot_unstable`、`decrypt_failed` 或 `integrity_failed` 中的具体原因。
3. `session` 失败时，汇总结果拒绝消息导出。
4. 失败事件只包含相对路径和分类原因，不包含密钥或聊天内容。

## 成功标准

- 辅助数据库失败不会阻断已验证消息库的导出与上传。
- 必需数据库失败绝不触发消息上传。
- 监控页能区分必需库失败与辅助库降级，并显示可操作的分类原因。
- 现有 V4 目录发现和临时明文清理行为保持不变。
